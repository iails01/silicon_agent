# 任务分支自动生成设计（方案 A）

## 背景
当前任务创建要求用户手动输入 `target_branch`。这导致：
- 用户输入成本高、易出错。
- 多入口（单任务、批量）分支命名不一致。
- 分支冲突策略不统一。

目标是改为后端统一自动生成分支名，前端仅展示规则，不再要求手填。

## 目标
1. 任务创建时自动生成 `target_branch`。
2. 命名规则：`silicon_agent/<task_id末6位>`。
3. 冲突时自动递增后缀：`-2/-3/...`。
4. 前端去掉“目标分支必填”。

## 非目标
1. 不修改工作区创建/推送主流程语义。
2. 不新增“系统分支策略配置中心”。
3. 不引入跨项目全局唯一分支强约束（默认按项目维度判断冲突）。

## 方案概述
采用后端集中生成（方案 A）：
- 前端不再手填分支；仅提示自动规则。
- 后端在 `TaskService` 创建任务时生成并写入 `task.target_branch`。
- 批量创建复用同一规则，并处理同事务内冲突。
- 执行层继续使用 `task.target_branch`，无需改动调用协议。

## 详细设计

### 1) 后端生成时机
在 `create_task` 中：
- 先 `session.add(task)` 并 `flush()`，确保拿到 `task.id`。
- 再依据 `task.id` 生成候选分支并回写 `task.target_branch`。
- 最后提交事务。

批量创建同理：每个任务先 `flush()` 拿到 ID，再生成目标分支。

### 2) 命名规则
- 基础名：`silicon_agent/<last6>`
- `<last6>` 来自 `task.id` 的末 6 位（例如 `...a1b2c3` -> `a1b2c3`）。
- 分支名全部小写（UUID 默认小写，无额外转换成本）。

### 3) 冲突处理
冲突策略已确认：自动追加序号。
- 首选：`silicon_agent/<last6>`
- 冲突：`silicon_agent/<last6>-2`
- 再冲突：`...-3` 递增

冲突判定建议默认按 `project_id + target_branch`：
- 同项目内保证唯一。
- `project_id` 为空时，按全局判重。

同时在批量创建单事务中维护“本次已分配候选集合”，避免同批未提交任务相互撞名。

### 4) 前端行为
任务创建页：
- 移除 `target_branch` 输入必填校验。
- 不再提交用户输入 `target_branch`。
- 增加只读提示：
  - `分支将自动生成：silicon_agent/<taskId后6位>（冲突自动追加序号）`

### 5) 兼容性
- `TaskCreateRequest.target_branch` 字段保留（兼容已有客户端），但服务端创建时忽略它。
- 返回结构保留 `target_branch`，值为系统生成结果。

## 错误处理
1. 后缀重试上限（例如 200 次）后仍冲突：创建失败并返回明确错误。
2. 生成逻辑异常（极端情况）：创建失败，错误写入响应 detail。

## 测试计划

### 后端测试
1. 创建任务会自动生成 `target_branch`（且忽略请求中的手填值）。
2. 同项目存在冲突时自动生成 `-2/-3`。
3. 批量创建同事务内分支不重复。
4. 超过重试上限时返回明确错误。

### 前端验收
1. 单任务创建不再要求输入分支。
2. 批量创建不再要求输入分支。
3. 创建成功后在任务详情可看到自动生成的 `target_branch`。

## 风险与缓解
1. 历史任务已有手工命名：不影响，仅新建任务启用自动策略。
2. 并发冲突：通过查询+同事务已分配集合+重试上限降低风险。
3. 认知偏差：前端显式展示自动规则，减少“为何不是手填分支”的疑惑。

## 实施顺序
1. 后端先落地自动生成与冲突处理。
2. 补后端测试。
3. 前端移除手填依赖并加提示文案。
4. 联调验收（单任务/批量/详情展示）。
